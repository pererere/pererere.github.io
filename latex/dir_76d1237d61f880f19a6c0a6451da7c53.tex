Socket.\+IO enables real-\/time bidirectional event-\/based communication. It consists of\+:


\begin{DoxyItemize}
\item a Node.\+js server (this repository)
\item a \href{https://github.com/socketio/socket.io-client}{\texttt{ Javascript client library}} for the browser (or a Node.\+js client)
\end{DoxyItemize}

\end{DoxyItemize}

Its main features are\+:\hypertarget{Readme.md_autotoc_md19414}{}\doxysubsubsubsection{\texorpdfstring{Reliability}{Reliability}}\label{Readme.md_autotoc_md19414}
Connections are established even in the presence of\+:
\begin{DoxyItemize}
\item proxies and load balancers.
\item personal firewall and antivirus software.
\end{DoxyItemize}

For this purpose, it relies on \href{https://github.com/socketio/engine.io}{\texttt{ Engine.\+IO}}, which first establishes a long-\/polling connection, then tries to upgrade to better transports that are "{}tested"{} on the side, like Web\+Socket. Please see the \href{https://github.com/socketio/engine.io\#goals}{\texttt{ Goals}} section for more information.\hypertarget{Readme.md_autotoc_md19415}{}\doxysubsubsubsection{\texorpdfstring{Auto-\/reconnection support}{Auto-\/reconnection support}}\label{Readme.md_autotoc_md19415}
Unless instructed otherwise a disconnected client will try to reconnect forever, until the server is available again. Please see the available reconnection options \href{https://socket.io/docs/v3/client-api/\#new-Manager-url-options}{\texttt{ here}}.\hypertarget{Readme.md_autotoc_md19416}{}\doxysubsubsubsection{\texorpdfstring{Disconnection detection}{Disconnection detection}}\label{Readme.md_autotoc_md19416}
A heartbeat mechanism is implemented at the Engine.\+IO level, allowing both the server and the client to know when the other one is not responding anymore.

That functionality is achieved with timers set on both the server and the client, with timeout values (the {\ttfamily ping\+Interval} and {\ttfamily ping\+Timeout} parameters) shared during the connection handshake. Those timers require any subsequent client calls to be directed to the same server, hence the {\ttfamily sticky-\/session} requirement when using multiples nodes.\hypertarget{Readme.md_autotoc_md19417}{}\doxysubsubsubsection{\texorpdfstring{Binary support}{Binary support}}\label{Readme.md_autotoc_md19417}
Any serializable data structures can be emitted, including\+:


\begin{DoxyItemize}
\item \href{https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/ArrayBuffer}{\texttt{ Array\+Buffer}} and \href{https://developer.mozilla.org/en-US/docs/Web/API/Blob}{\texttt{ Blob}} in the browser
\item \href{https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/ArrayBuffer}{\texttt{ Array\+Buffer}} and \href{https://nodejs.org/api/buffer.html}{\texttt{ Buffer}} in Node.\+js
\end{DoxyItemize}\hypertarget{Readme.md_autotoc_md19418}{}\doxysubsubsubsection{\texorpdfstring{Simple and convenient API}{Simple and convenient API}}\label{Readme.md_autotoc_md19418}

\hypertarget{Readme.md_autotoc_md19419}{}\doxysubsubsubsection{\texorpdfstring{Cross-\/browser}{Cross-\/browser}}\label{Readme.md_autotoc_md19419}
Browser support is tested in Sauce Labs\+:

\href{https://saucelabs.com/u/socket}{\texttt{ }}\hypertarget{Readme.md_autotoc_md19420}{}\doxysubsubsubsection{\texorpdfstring{Multiplexing support}{Multiplexing support}}\label{Readme.md_autotoc_md19420}
In order to create separation of concerns within your application (for example per module, or based on permissions), Socket.\+IO allows you to create several {\ttfamily Namespaces}, which will act as separate communication channels but will share the same underlying connection.\hypertarget{Readme.md_autotoc_md19421}{}\doxysubsubsubsection{\texorpdfstring{Room support}{Room support}}\label{Readme.md_autotoc_md19421}
Within each {\ttfamily Namespace}, you can define arbitrary channels, called {\ttfamily Rooms}, that sockets can join and leave. You can then broadcast to any given room, reaching every socket that has joined it.

This is a useful feature to send notifications to a group of users, or to a given user connected on several devices for example.

{\bfseries{Note\+:}} Socket.\+IO is not a Web\+Socket implementation. Although Socket.\+IO indeed uses Web\+Socket as a transport when possible, it adds some metadata to each packet\+: the packet type, the namespace and the ack id when a message acknowledgement is needed. That is why a Web\+Socket client will not be able to successfully connect to a Socket.\+IO server, and a Socket.\+IO client will not be able to connect to a Web\+Socket server (like {\ttfamily ws\+://echo.websocket.\+org}) either. Please see the protocol specification \href{https://github.com/socketio/socket.io-protocol}{\texttt{ here}}.\hypertarget{Readme.md_autotoc_md19422}{}\doxysubsection{\texorpdfstring{Installation}{Installation}}\label{Readme.md_autotoc_md19422}

\hypertarget{Readme.md_autotoc_md19426}{}\doxysubsubsection{\texorpdfstring{In conjunction with Express}{In conjunction with Express}}\label{Readme.md_autotoc_md19426}
Starting with {\bfseries{3.\+0}}, express applications have become request handler functions that you pass to {\ttfamily http} or {\ttfamily http} {\ttfamily Server} instances. You need to pass the {\ttfamily Server} to {\ttfamily socket.\+io}, not the express application function. Also make sure to call {\ttfamily .listen} on the {\ttfamily server}, not the {\ttfamily app}.


\begin{DoxyCode}{0}
\DoxyCodeLine{const\ app\ =\ require('express')();}
\DoxyCodeLine{const\ server\ =\ require('http').createServer(app);}
\DoxyCodeLine{const\ io\ =\ require('socket.io')(server);}
\DoxyCodeLine{io.on('connection',\ ()\ =>\ \{\ /*\ …\ */\ \});}
\DoxyCodeLine{server.listen(3000);}

\end{DoxyCode}
