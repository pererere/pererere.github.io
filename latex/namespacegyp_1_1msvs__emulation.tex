{\footnotesize\ttfamily \label{namespacegyp_1_1msvs__emulation_a28e813ee1be02285cdc5437040e76444} 
gyp.\+msvs\+\_\+emulation.\+Generate\+Environment\+Files (\begin{DoxyParamCaption}\item[{}]{toplevel\+\_\+build\+\_\+dir}{, }\item[{}]{generator\+\_\+flags}{, }\item[{}]{system\+\_\+includes}{, }\item[{}]{open\+\_\+out}{}\end{DoxyParamCaption})}

\begin{DoxyVerb}It's not sufficient to have the absolute path to the compiler, linker,
etc. on Windows, as those tools rely on .dlls being in the PATH. We also
need to support both x86 and x64 compilers within the same build (to support
msvs_target_platform hackery). Different architectures require a different
compiler binary, and different supporting environment variables (INCLUDE,
LIB, LIBPATH). So, we extract the environment here, wrap all invocations
of compiler tools (cl, link, lib, rc, midl, etc.) via win_tool.py which
sets up the environment, and then we do not prefix the compiler with
an absolute path, instead preferring something like "cl.exe" in the rule
which will then run whichever the environment setup has put in the path.
When the following procedure to generate environment files does not
meet your requirement (e.g. for custom toolchains), you can pass
"-G ninja_use_custom_environment_files" to the gyp to suppress file
generation and use custom environment files prepared by yourself.\end{DoxyVerb}
